## Perspective 1 — I Am the API Key

I do not exist yet. But KRMC's CTO just visited `resumeproof.com/docs` and decided to integrate. Here is my origin story.

---

### Chapter 1 — Client Registration

#### Step 1.1 — The Sign-Up Request

The CTO fills in the registration form on `resumeproof.com/register`:
- Organization name: KRMC Placement Portal
- Email: cto@krmc.edu
- Plan: Free (500 verifications/month)

Their browser POSTs to:
```
POST https://api.resumeproof.com/v1/auth/register
Content-Type: application/json

{
  "orgName": "KRMC Placement Portal",
  "email": "cto@krmc.edu",
  "password": "their_secure_password"
}
```

**File:** `ms1-orchestrator/src/routes/authRoutes.ts`
**Route:** `POST /v1/auth/register`
**Controller:** `authController.register(req, res, next)`
**File:** `ms1-orchestrator/src/controllers/authController.ts`

The controller calls:
**File:** `ms1-orchestrator/src/services/clientService.ts`
**Function:** `registerNewClient(orgName, email, password)`

What happens inside `registerNewClient`:

1. **Check for duplicate email**
   **File:** `ms1-orchestrator/src/repositories/clientRepository.ts`
   **Function:** `findClientByEmail(email)`
   If found → throw `AppError('email_already_registered', 409)`

2. **Hash the password**
   Using `bcrypt.hash(password, 10)`. The raw password is immediately discarded from memory. Only the hash is stored.
   **Package:** `bcrypt`

3. **Write the client row**
   **File:** `ms1-orchestrator/src/repositories/clientRepository.ts`
   **Function:** `createClient(data)`
   Inserts into `clients` table:
   ```
   id: uuid (generated by DB)
   name: "KRMC Placement Portal"
   email: "cto@krmc.edu"
   passwordHash: "$2b$10$..."
   plan: "free"
   monthlyQuota: 500
   webhookUrl: null       (set later)
   webhookSecret: null    (set later)
   isVerified: false      (email not confirmed yet)
   createdAt: now()
   ```

4. **Send verification email**
   **File:** `ms1-orchestrator/src/services/emailService.ts`
   **Function:** `sendEmailVerificationLink(email, verificationToken)`
   
   A `verificationToken` is generated with `crypto.randomBytes(32).toString('hex')`. It is stored (hashed) in the `email_verifications` table with a 24-hour expiry. The raw token goes in the email link:
   `https://resumeproof.com/verify-email?token=abc123...`
   
   **Package:** Node.js built-in `crypto`
   **Package:** `nodemailer` — SMTP email sending (via SendGrid or AWS SES free tier)

5. **Return 201** with `{ message: "Check your email to complete registration" }`

**No API key yet.** I still don't exist.

#### Step 1.2 — Email Verification

The CTO clicks the link in their email. Browser hits:
```
GET https://api.resumeproof.com/v1/auth/verify-email?token=abc123...
```

**File:** `ms1-orchestrator/src/controllers/authController.ts`
**Function:** `verifyEmail(req, res, next)`

Calls:
**File:** `ms1-orchestrator/src/services/clientService.ts`
**Function:** `verifyClientEmail(rawToken)`

1. Hash the incoming token: `crypto.createHash('sha256').update(rawToken).digest('hex')`
2. Look up the `email_verifications` table by hash, check it hasn't expired
3. Mark `clients.isVerified = true`
4. Delete the verification row (single-use token)
5. Redirect to dashboard: `https://resumeproof.com/dashboard`

**Package:** Node.js built-in `crypto`

### Chapter 2 — I Am Born (API Key Generation)

#### Step 2.1 — The Generate Key Request

The CTO is now in the dashboard. They click "Generate API Key." Their browser (authenticated with a session JWT) POSTs:

```
POST https://api.resumeproof.com/v1/keys/generate
Authorization: Bearer eyJhbGci...
Content-Type: application/json

{
  "label": "Production Key",
  "environment": "live"
}
```

**File:** `ms1-orchestrator/src/routes/keyRoutes.ts`
**Route:** `POST /v1/keys/generate`

This route uses a **different auth middleware** — `authenticateJwt` instead of `authenticateApiKey`. Dashboard routes are protected by JWT session tokens, not API keys.

**File:** `ms1-orchestrator/src/middlewares/authenticateJwt.ts`
**Function:** `authenticateJwt(req, res, next)`

Verifies the JWT using `jsonwebtoken.verify(token, JWT_SECRET)`. Extracts `clientId` from the payload. Attaches to `req.clientId`.

**Package:** `jsonwebtoken`

The controller calls:
**File:** `ms1-orchestrator/src/services/apiKeyService.ts`
**Function:** `generateApiKey(clientId, label, environment)`

**Step 1 — Generate the raw key**

```
prefix = environment === 'live' ? 'rp_live_' : 'rp_test_'
randomPart = crypto.randomBytes(24).toString('base64url')   // 32 chars, URL-safe
rawKey = prefix + randomPart
// Result: "rp_live_K9mX2qL8nR4vT7wYaB3cP5dE1fG0h"
```

**Package:** Node.js built-in `crypto`

**Step 2 — Hash the raw key**

```
keyHash = bcrypt.hashSync(rawKey, 10)
```

The hash goes in the database. The raw key is shown to the user **once and never again**. After this function returns, the raw key exists only in this function's local scope and in the HTTP response. It is not logged anywhere.

**Package:** `bcrypt`

**Step 3 — Write to database**
**File:** `ms1-orchestrator/src/repositories/apiKeyRepository.ts`
**Function:** `createApiKey(clientId, keyHash, prefix, label)`

Inserts into `api_keys`:
```
id: uuid
clientId: "uuid-of-krmc"
keyHash: "$2b$10$..."       ← the hash
prefix: "rp_live_"          ← for display and lookup narrowing
label: "Production Key"
isActive: true
lastUsed: null
createdAt: now()
```

**Step 4 — Return the raw key in the response (ONCE)**

```json
{
  "apiKey": "rp_live_K9mX2qL8nR4vT7wYaB3cP5dE1fG0h",
  "prefix": "rp_live_",
  "label": "Production Key",
  "createdAt": "2026-07-16T10:00:00Z",
  "warning": "Store this key securely. It will not be shown again."
}
```

I now exist. I am `rp_live_K9mX2qL8nR4vT7wYaB3cP5dE1fG0h`.

The CTO copies me and stores me in their server's `.env` file:
```
RESUMEPROOF_API_KEY=rp_live_K9mX2qL8nR4vT7wYaB3cP5dE1fG0h
```

The dashboard will from now on only show my prefix: `rp_live_••••••••••••••••••••••••` — the rest is gone from their systems too. If they lose me, they generate a new key. I cannot be recovered.

### Chapter 3 — Webhook Configuration

Before making their first API call, the CTO configures where our system should deliver results.

```
POST https://api.resumeproof.com/v1/settings/webhook
Authorization: Bearer eyJhbGci...

{
  "webhookUrl": "https://krmc.edu/api/resumeproof/callback"
}
```

**File:** `ms1-orchestrator/src/controllers/settingsController.ts`
**Function:** `updateWebhookSettings(req, res, next)`

Calls:
**File:** `ms1-orchestrator/src/services/clientService.ts`
**Function:** `setWebhookConfig(clientId, webhookUrl)`

**Step 1 — Validate the webhook URL**

`zod` validates it as a valid HTTPS URL. HTTP (non-TLS) is rejected — we won't send signed payloads over unencrypted connections.

**Step 2 — Generate the webhook secret**

```
webhookSecret = crypto.randomBytes(32).toString('hex')
// Result: "d4f8a2c1e9b3f7a0c2d5e8f1a4b7c0d3e6f9a2b5c8d1e4f7a0b3c6d9e2f5a8b1"
```

This is the shared secret used to sign our outgoing webhook payloads. KRMC stores this in their `.env` as `RESUMEPROOF_WEBHOOK_SECRET`. We store it (encrypted at rest via PostgreSQL column encryption or application-level encryption before insert) in `clients.webhookSecret`.

**Step 3 — Test the endpoint**

**File:** `ms1-orchestrator/src/services/webhookService.ts`
**Function:** `sendWebhookTestPing(webhookUrl, webhookSecret)`

Sends a POST to KRMC's endpoint with a test payload:
```json
{
  "event": "webhook.test",
  "message": "ResumeProof webhook successfully connected",
  "timestamp": "2026-07-16T10:05:00Z"
}
```

Signed with the new secret. If KRMC's endpoint returns 2xx → webhook config is saved. If not → 400 "Webhook endpoint unreachable."

**Step 4 — Update client record**
**File:** `ms1-orchestrator/src/repositories/clientRepository.ts`
**Function:** `updateWebhookConfig(clientId, webhookUrl, encryptedSecret)`

**Package:** Node.js built-in `crypto`

**Response:**
```json
{
  "webhookUrl": "https://krmc.edu/api/resumeproof/callback",
  "webhookSecret": "d4f8a2c1...",
  "warning": "Store this secret. It will not be shown again."
}
```

### Chapter 4 — The Integration

#### Step 4.1 — KRMC's Developer Reads the Docs

The docs tell them: your backend calls our API. Your API key lives on your server. Your frontend never touches our API. Here is what to build.

**What KRMC builds on their side:**

```
krmc-server/
├── src/
│   ├── routes/
│   │   ├── applyRoute.ts               POST /api/apply  (receives form from student)
│   │   └── resumeProofWebhookRoute.ts  POST /api/resumeproof/callback
│   ├── services/
│   │   └── resumeProofClient.ts        submitForVerification(), pollForResult()
│   └── utils/
│       └── webhookVerifier.ts          verifyWebhookSignature()
```

**Package KRMC installs:** `axios` for HTTP calls to our API, `form-data` for multipart, Node.js `crypto` for webhook verification.

#### Step 4.2 — The First Real API Call

A candidate submits their application on `krmc.edu/apply`. KRMC's server receives the form at their `POST /api/apply` route. Their `handleApplicationSubmit()` function calls their `resumeProofClient.submitForVerification()`.

Their function constructs:

```
POST https://api.resumeproof.com/v1/verify
x-api-key: rp_live_K9mX2qL8nR4vT7wYaB3cP5dE1fG0h
Content-Type: multipart/form-data

Fields:
  resume:           <PDF binary>
  githubUrl:        "https://github.com/arjun-mehra"
  jobDescription:   "Backend Engineer with Node.js..."   (optional)
  candidateEmail:   "arjun@iit.ac.in"                   (optional, for their own records)
```

**Our response (202 Accepted, in ~200ms):**
```json
{
  "transactionId": "RPV-2026-07-16-A4X9F2",
  "status": "processing",
  "estimatedSeconds": 45,
  "pollUrl": "/v1/verify/RPV-2026-07-16-A4X9F2"
}
```

KRMC stores `transactionId` in their own `applications` table against Arjun's record. They return `{ message: "Application submitted. Results pending." }` to the student's browser. The student's browser is now out of the picture.

#### Step 4.3 — KRMC Waits (Two Options)

**Option A — Webhook (recommended)**

KRMC does nothing. They wait. In ~45 seconds, we POST the result to `https://krmc.edu/api/resumeproof/callback`.

Their webhook handler in `resumeProofWebhookRoute.ts`:

1. Reads raw request body as a string (before JSON parsing — HMAC is over raw bytes)
2. Calls their `webhookVerifier.verifyWebhookSignature(rawBody, signature, secret)`
3. Which does: `expected = 'sha256=' + crypto.createHmac('sha256', secret).update(rawBody).digest('hex')`
4. Compares with `crypto.timingSafeEqual()` — timing-safe comparison
5. If valid: parses JSON, looks up `transactionId` in their DB, updates candidate status
6. Returns `200 { received: true }` — immediately, before any DB work

**Option B — Polling (for showing live progress to a hiring manager)**

KRMC's dashboard makes periodic requests:
```
GET https://api.resumeproof.com/v1/verify/RPV-2026-07-16-A4X9F2
x-api-key: rp_live_K9mX2qL8nR4vT7wYaB3cP5dE1fG0h
```

Response while processing:
```json
{ "status": "processing", "transactionId": "RPV-2026-07-16-A4X9F2" }
```
→ HTTP 202

Response when done:
```json
{
  "status": "done",
  "transactionId": "RPV-2026-07-16-A4X9F2",
  "result": {
    "verificationStatus": "verified",
    "confidenceScore": 91,
    "github": { "username": "arjun-mehra", "reposFound": 12, "verifiedProjects": 4 },
    "skillAlignment": 87,
    "matchedSkills": ["Node.js", "PostgreSQL"],
    "missingSkills": ["Kubernetes"],
    "flags": []
  }
}
```
→ HTTP 200

## The API Key Security Model — Complete Picture

```
What exists in our DB:    keyHash (bcrypt, cost 10)
What the client sees:     rp_live_K9mX2... (once, at generation)
What the client stores:   rp_live_K9mX2... (in their .env)
What we show on dashboard: rp_live_•••••••••••••••••••
What travels in requests: rp_live_K9mX2... (in x-api-key header, over HTTPS)
What NGINX sees:          the raw key (used for rate limiting by key value)
What Express sees:        the raw key (hashed and compared against DB)
What logs contain:        NEVER the raw key — only the prefix "rp_live_"
What happens if DB leaks: attacker has bcrypt hashes — computationally infeasible to reverse
What happens if key leaks: client revokes it via dashboard (DELETE /v1/keys/:id), generates new
```

The only way an attacker can use a leaked key is if they have the raw key string itself — not the hash. So the risk is the client's `.env` file, not our database.

## The Quota Enforcement Flow

Every call to `POST /v1/verify` passes through:

**File:** `ms1-orchestrator/src/middlewares/enforceQuotaMiddleware.ts`
**Function:** `enforceMonthlyQuota(req, res, next)`

1. Get today's date in `YYYY-MM` format as the Redis key suffix
2. `INCR rp:quota:{clientId}:2026-07` in Redis (atomic increment)
3. If result > client's `monthlyQuota` (from `req.client.monthlyQuota`): return `429 { error: "quota_exceeded", resetDate: "2026-08-01" }`
4. Set Redis key TTL to `end of current month + 7 days` (auto-cleanup)
5. Call `next()`

This is checked in Redis (not PostgreSQL) for speed — quota enforcement happens on every request, needs sub-millisecond response.

**Package:** `ioredis` — atomic Redis INCR command.

## The JWT Strategy

Dashboard routes (key management, settings, usage stats) use JWT — not API keys. API keys are for machine-to-machine. JWTs are for human dashboard sessions.

**Access token:** 1 hour lifetime, signed with `JWT_SECRET` env var using `jsonwebtoken.sign()`. Payload: `{ clientId, email, plan }`.

**Refresh token:** 30 day lifetime. Stored as a hash in the `sessions` table. When the access token expires, the client sends the refresh token to `POST /v1/auth/refresh`, we verify the hash, issue a new access token.

**Token rotation:** Every refresh call issues a new refresh token and invalidates the old one (updates the hash in `sessions`). This means a stolen refresh token can only be used once.

**Package:** `jsonwebtoken`

```

─────────────────────────

  → clientService.registerNewClient()

  → clientService.setWebhookConfig()
    → webhookService.sendTestPing()
    → clientRepository.updateWebhookConfig()[updates: clients table]
    → returns webhookSecret ONCE

[KRMC backend calls us]
```
